# XyzyBlog 后台模块补全设计

## 目标

在保留现有前台博客 API 的基础上，补全可独立运行的 `admin` 后台 REST API，使项目覆盖课件要求的文章、分类、标签、友链、用户、角色和菜单权限管理，并能够通过 Maven 构建、自动化测试及真实 MySQL/Redis 冒烟测试。

本次不开发 Vue、React 或静态网页。

## 现状

- `framework` 已包含前台业务使用的实体、Mapper、Service、安全和公共工具。
- `blog` 已实现文章展示、分类、友链、评论、注册登录、用户信息、日志及浏览量同步。
- `admin` 只有空目录和 Maven 配置，没有启动类、配置或接口。
- 数据库已有文章、分类、评论、友链和用户表，但缺少标签、角色、菜单及关联表。
- 根项目构建在 `admin` 的 Spring Boot 重打包阶段失败，因为找不到启动类。
- MySQL 服务可用；Redis 尚未运行。

## 模块架构

### framework

继续作为共享业务层，新增后台需要的实体、DTO、VO、Mapper 和 Service。共享逻辑只实现一次，`blog` 与 `admin` 通过模块依赖调用。现有公开方法和前台响应结构保持兼容。

### blog

保留端口 `7777`、现有路由和行为。仅在共享代码调整影响编译或正确性时做必要兼容修改，不改变已有接口路径。

### admin

新增独立 Spring Boot 启动类和 `application.yml`，默认端口 `8989`。模块包含后台 Controller、安全配置、JWT 过滤器和必要的模块配置。后台不复制共享业务实现。

## 数据模型

在现有 `sql/init.sql` 中补充：

- `t_tag`：标签信息。
- `t_article_tag`：文章与标签的多对多关系。
- `t_role`：角色信息及状态。
- `t_menu`：目录、菜单、按钮权限和权限标识。
- `t_user_role`：用户与角色关系。
- `t_role_menu`：角色与菜单权限关系。

为现有数据补充管理员角色、基础菜单权限及关联记录。数据库变更必须可重复执行，避免重复初始化导致失败。

## 认证与授权

- 前后台沿用 Spring Security、JWT 和 Redis 登录会话。
- 后台提供独立登录、退出和当前用户信息接口。
- 登录成功后返回令牌；请求通过 JWT 过滤器解析用户并从 Redis 恢复会话。
- 后台账号必须状态正常，并具备后台角色或菜单权限。
- Controller 使用权限标识保护管理操作；无认证返回统一未登录响应，无权限返回统一禁止访问响应。
- 密码只保存 BCrypt 哈希，不在响应、日志或测试报告中输出明文。

## 后台接口范围

### 登录与当前用户

- 登录、退出。
- 查询当前用户、角色和可访问菜单。

### 内容管理

- 文章：分页检索、详情、新增、修改、删除、发布状态管理。
- 分类：分页或列表查询、新增、修改、删除、状态管理。
- 标签：分页或列表查询、新增、修改、删除。
- 友链：分页查询、新增、修改、删除、审核状态管理。

### 系统管理

- 用户：分页检索、新增、修改、删除、状态管理、分配角色。
- 角色：分页检索、新增、修改、删除、状态管理、分配菜单权限。
- 菜单：树形查询、新增、修改、删除。

删除操作优先沿用 MyBatis-Plus 逻辑删除；关联表在事务中同步维护。被引用的数据不能直接删除时返回明确业务错误。

## 数据流

请求进入 `admin` Controller 后先经过 Spring Security 和 JWT 过滤器，再调用 `framework` Service。Service 负责参数规则、权限相关业务、事务和关联表维护，通过 Mapper 访问 MySQL，并按需读写 Redis。所有接口最终返回统一 `ResponseResult`。

前台文章查询继续使用 Redis 浏览量。后台修改文章后不会覆盖 Redis 中更新更快的浏览量；删除或状态变更时同步处理相关缓存。

## 校验与异常

- 对请求 DTO 使用 Jakarta Bean Validation。
- 校验用户名、昵称、邮箱、分类名、标签名和角色名等唯一字段。
- 对不存在、已删除、状态异常或被引用的资源给出明确业务响应。
- 使用现有全局异常处理机制，补齐参数校验、请求格式和数据约束异常。
- 不向客户端泄露 SQL、堆栈、数据库密码或内部路径。

## 测试策略

### 自动化测试

- Service 单元测试覆盖权限分配、关联更新、唯一性校验和关键状态转换。
- Controller 测试覆盖登录要求、权限拒绝、参数校验和主要 CRUD 响应。
- 保留测试隔离，不依赖开发数据库中的历史数据。

### 构建验证

- `mvn clean test` 全部通过。
- `mvn clean package` 全部模块通过，并生成可运行的 `blog`、`admin` JAR。

### 真实环境冒烟测试

- 使用本机 MySQL 初始化 `myblog`。
- 启动本机或容器 Redis。
- 分别启动 `blog:7777` 和 `admin:8989`。
- 验证前台公开查询、用户登录、评论，以及后台登录和每类管理资源的查询、新增、修改、删除主路径。
- 保存脱敏后的命令、状态码和测试结论到测试报告。
